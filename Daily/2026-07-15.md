### 18:15 — Host crashed/rebooted, Plex + Tdarr down (NVIDIA driver gone). RESOLVED.
Rob flagged the box had crashed and Plex was offline (also couldn't reach Uptime Kuma). Investigated:
- **Uptime Kuma was never actually down** — CT 115 running, `/dashboard` 200. Just post-boot timing; Rob hit it before it came up. No action needed.
- **Plex (CT 100) + Tdarr (CT 109) were stopped** — both failed to autostart with `Device /dev/nvidia-* does not exist`.
- **Root cause:** the NVIDIA driver (580.173.02, hand-installed via `.run` on 30 Jun against kernel `7.0.12-1-pve`) had NO module for the now-running kernel `7.0.14-4-pve`. apt auto-upgraded proxmox-kernel three times since (7.0.14-2/-3/-4) and the crash rebooted onto -4. The `.run` installer doesn't rebuild on kernel changes and there was no DKMS. So `modprobe nvidia` → module not found → no GPU → Plex/Tdarr (both GPU passthrough) can't start.
- **Durable fix (so this never recurs):** installed `dkms` + `proxmox-headers-7.0.14-4-pve` + `proxmox-default-headers`, re-ran the on-host installer with `--dkms` (`dkms status` now `nvidia/580.173.02, 7.0.14-4-pve: installed` → auto-rebuilds on every future kernel bump). Created + enabled a boot-time oneshot `nvidia-lxc-devices.service` (host) that runs `nvidia-modprobe -c 0 -u -m` **before** `pve-guests.service`, so all device nodes (nvidia0/ctl/uvm/uvm-tools/modeset) exist before the containers autostart. Documented in `Context/Infrastructure.md` (new GPU section).
- Started both CTs: Plex serving 200 locally + via `https://plex.cracky.co.uk`; Tdarr running.
- **Thin pool bonus:** post-reboot `pct fstrim` across all CTs reclaimed ~20 points (96.99% → 77.29%) — the opposite of this morning's "trim is tapped out". The big reclaims were freshly-rebooted CTs (Plex 15.5G, jarvis 22.8G, paperless 22.5G, plausible 16.8G) shedding accumulated untrimmed deleted blocks. So a reboot + trim buys real headroom; the standing growth-pressure proposal still stands but the pool isn't at crisis right now.
