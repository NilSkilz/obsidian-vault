### Infra: vaultwarden 502 — IP conflict on .17 (fixed)
Rob flagged vault.cracky.co.uk showing down in Kuma. **Root cause: an IP conflict.** The vaultwarden CT (113, static .17) was healthy on the LAN (200 on :8080/alive), but NPM 502'd because **an iPhone had grabbed a DHCP lease on 192.168.1.17** — the same IP as the container. NPM's ARP cache resolved .17 to the phone's MAC (`72:3b:a0:f9:77:29`) instead of the CT's (`bc:24:11:39:42:02`), so traffic went to a device that doesn't answer :8080 → bad gateway.
- **Why it happened:** the UDM DHCP pool is **`.6`–`.254`**, overlapping every static service IP. The old infra note ("pool starts above the statics") was **wrong**. The newer CTs (.16 MC, .17 vault, .18 paperless, .19 kuma) had **no DM reservation**, only their Proxmox static — so the UDM freely leased .17 to a phone. Same failure mode as Seerr in July, just triggered by a colliding DHCP client instead of drift.
- **Fix:** added DM fixed-IP reservations for all four unprotected CTs via the UniFi API (writes work now — PUT `rest/user/<_id>`, contra the old "301 wall" note). The phone clung to its 24h lease so the reservation alone didn't bump it; `block-sta`+`unblock-sta` on the phone forced a full re-DHCP → it moved to **.21**, freeing .17. Flushed NPM's stale ARP to re-resolve. **Vault verified solidly 200 end-to-end**, .17 now held by the CT alone.
- **Durable lesson written to Infrastructure.md:** a Proxmox static inside the DHCP pool without a matching DM reservation is a time bomb; belt-and-braces needs BOTH.
